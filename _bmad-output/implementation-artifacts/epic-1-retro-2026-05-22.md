## Challenges & Growth Areas

Amelia (Developer): "Maintenant les zones de friction. Rien de critique, mais des patterns à corriger pour Epic 2."

**1. Dev Notes API inaccuracies required runtime correction (3 occurrences)**

| Story | Dev Notes claim | Reality | Fix applied |
|-------|----------------|---------|-------------|
| 1.1 | `setuptools.backends.legacy:build` | Doesn't exist — use `setuptools.build_meta` | Fixed at dev time |
| 1.4 | `benchmark.stats.mean` (pytest-benchmark) | `benchmark.stats.stats.mean` (5.x API) | Fixed at dev time |
| 1.5 | `physlink.__version__` accessible | `__version__` not defined in `__init__.py` | Fixed at review time |

Charlie (Senior Dev): "Trois fois sur six stories, on a eu des Dev Notes qui donnaient une API inexacte. Heureusement, les dev agents ont adapté — mais ça génère de l'incertitude inutile."

**Root cause:** Dev Notes are written at planning time against expected APIs, but library versions and actual module exports evolve. No verification step exists between planning and implementation.

**Action item:** Add an explicit "Verify API version" step in story Dev Notes for any non-stdlib import used. Document the verified version in the story completion notes.

**2. TestSrcLayoutEnforcement failure in CI — unexpected interaction**

The `src/` layout tests in `test_toolchain_compliance.py` (Story 1.1) were designed to verify that `import physlink` fails from the repo root. But in CI, physlink is installed with `pip install -e ".[dev]"`, making the import succeed — causing the tests to fail.

This required adding `@pytest.mark.skipif(os.getenv("CI") == "true" or importlib.util.find_spec("physlink") is not None, ...)` to the affected tests.

Dana (QA Engineer): "C'est un cas typique de test dont la prémisse change selon l'environnement d'exécution. Le correctif est bon, mais on aurait dû le prévoir dès Story 1.1."

**Root cause:** Story 1.1 was greenfield and didn't anticipate the CI editable install. Future stories that test environment-specific behavior should include a CI environment note.

**3. YOUR-ORG placeholder left in README/CI badge URLs**

Story 1.6 left `YOUR-ORG` in `README.md` and CI badge URLs because no git remote `origin` was configured at dev time. This means:
- The CI badge in README links to a non-existent repo URL
- The "Open in Colab" button links to a non-existent GitHub path
- The Colab notebook notebook quick-start cell has `YOUR-ORG` in CI badge links

Alice (Product Owner): "C'est la limitation la plus visible pour un utilisateur externe. Si quelqu'un clone le repo aujourd'hui, le README a des liens cassés."

**Resolution:** Must configure git remote `origin` before any external sharing. This is the #1 critical item for Epic 2 preparation.

## Key Insights

Amelia (Developer): "Voici les insights qui ont de la valeur pour la suite."

**Insight 1 — The BMAD framework dirs need ruff exclusions from day 1**

Story 1.1 discovered that `.agents/`, `.claude/`, `_bmad/`, `_bmad-output/` directories don't comply with physlink's strict ruff config. These must be excluded via `[tool.ruff] exclude` — this is a mandatory setup step for any BMAD + Python project.

**Insight 2 — Pre-commit and the git init dependency**

The Worldchain directory had no `.git` repo initially; `git init` was required for pre-commit to discover staged files. This should be documented as a project setup prerequisite.

**Insight 3 — `physlink.__version__` is undefined**

The module has no `__version__` attribute. The publish.yml smoke test needed to use the shell `$VERSION` variable instead. This is a gap: conventional Python packages expose `__version__` in `__init__.py`. This should be addressed in Epic 2's API finalization story (2.6).

**Insight 4 — The incremental `__all__` approach worked cleanly**

Building `__all__` symbol by symbol (one per story) and using `@pytest.mark.skip` guards that activate when the symbol is ready was an excellent pattern. The `test_api_stability.py` activation in Story 1.5 proved the pattern end-to-end.

Charlie (Senior Dev): "L'approche `@pytest.mark.skip` → activation progressive est exactement comment on devrait gérer les tests qui vérifient des symboles pas encore implémentés. À répliquer en Epic 2."
