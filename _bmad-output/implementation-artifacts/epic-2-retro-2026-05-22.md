## Challenges & Growth Areas

### 1. File List and Change Log incomplete at first version (pattern: 3 out of 6 stories)

Stories 2.1, 2.3, and 2.5 all had File Lists or Change Logs corrected at review time. This is a systematic process gap, not a one-off oversight.

**Root cause:** Completion Notes and File List are written at the end of implementation but before reviewing what was actually modified. Developers undercount files or forget to update Change Log with the actual state.

**Action item:** Add "File List complete + Change Log reflects actual state" as an explicit closing checklist item in story Dev Notes template.

### 2. YOUR-ORG — second consecutive epic without resolution

The critical debt item D-2 from Epic 1 was carried forward unresolved into Epic 2. The `YOUR-ORG` placeholder appears in README.md, mkdocs.yml, ci.yml, publish.yml, and notebooks/quickstart.ipynb. This means the CI badge shows a broken URL, the "Open in Colab" button links to a non-existent path, and the docs badge points nowhere.

**Impact for Epic 3:** If any Story 3.x references CI badge status or Colab notebook validation, the `YOUR-ORG` will cause false failures. Additionally, Story 3.6 (Export and Share Panel) validates Colab URL behavior — this interaction must be verified against a real remote.

**Resolution:** This must be configured BEFORE any Epic 3 story starts. It is the #1 critical path item.

### 3. mkdocs_autorefs cross-reference piège in docstring Examples

Story 2.5 hit a non-obvious bug: `info["bounds"][0]` in a Python docstring Example block was interpreted as a Markdown cross-reference to symbol `0` by `mkdocs_autorefs`, causing `mkdocs build --strict` to fail. Fix: replace `["bounds"][0]` with `len(info["bounds"])`.

**Pattern going forward:** Any docstring Example using dict subscript with an integer key (e.g., `result[0]`, `data["key"][0]`) risks this bug. Use `len()`, slices, or string keys that can't be confused with symbol references.

### 4. ruff ANN204 + mypy strict on special methods (Story 2.1)

`__iter__` required `-> Iterator[dict[str, Any]]` annotation to satisfy mypy ANN204. This was caught at review, not during development. The rule: **every special method (`__iter__`, `__len__`, `__repr__`, `__init__`) must have an explicit return type annotation** under mypy --strict.

## Key Insights

**Insight 1 — The bool-before-int guard is a systemic Python trap worth documenting once, referencing many times**

Stories 2.2 AND 2.3 both documented `isinstance(True, int) → True` as a critical constraint. Rather than re-explaining it, the pattern should be documented in a shared Dev Notes "Common Python Traps" reference. Story 3.x will also have type validation (e.g., DreamerV3Adapter step count) that will hit this same trap.

**Insight 2 — `--strict` mode in CI mkdocs build is the right call**

The `--strict` flag caught the `mkdocs_autorefs` bug before merge. Without it, the docs site would have deployed with a broken cross-reference silently. The flag should stay for all remaining epics.

**Insight 3 — The `test_docstring_completeness.py` AST approach is highly reusable**

Story 2.6 created an AST-based 28-test suite that walks all core modules and verifies Google-style docstring sections. This suite is automatically extensible — adding new public functions in Epics 3 and 4 will automatically be caught if they lack docstrings, without modifying the test suite.

**Insight 4 — Stories remain well-isolated — no cross-story interference**

Despite 6 stories all modifying the same files (`spaces.py`, `test_spaces.py`), there were zero merge conflicts and zero cross-story regressions. The deferred-implementation discipline (no stub methods, no preemptive exports) was the key.
