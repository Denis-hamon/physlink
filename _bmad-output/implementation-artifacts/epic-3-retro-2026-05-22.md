## Challenges & Growth Areas

### 1. HIGH issue in Story 3.4: import ordering in `load_checkpoint()` violated contract

The story spec explicitly required `_check_checkpoint_metadata(path)` as the **first call** in `load_checkpoint()`, before any deferred imports. The dev agent put `import torch`, `import os`, and `from safetensors.torch import load_file` before the metadata check. This would cause `ModuleNotFoundError` in torch-free environments instead of the documented `CheckpointCorruptError`.

**Root cause:** The "deferred imports" convention became so ingrained that the dev agent applied it at the top of every method body, without checking if the contract required something to run *before* the imports.

**Action item:** Add to Dev Notes template: "Methods with a 'pre-flight check' (metadata, guard clauses) must explicitly note WHAT runs before imports in the implementation plan."

### 2. `datetime.utcnow()` deprecation in two consecutive stories

Stories 3.4 and 3.5 both introduced `datetime.datetime.utcnow()`, which is deprecated in Python 3.12+. Story 3.4 was caught and fixed at review; Story 3.5 introduced it again and was caught at review again. The fix pattern (`datetime.datetime.now(datetime.timezone.utc)`) should be documented once and referenced.

**Pattern going forward:** Add to the "Previous Story Carry-Overs" section in each story's Dev Notes: "Use `datetime.datetime.now(datetime.timezone.utc)` — never `datetime.utcnow()` (deprecated in Python 3.12+)."

### 3. Sprint-status stuck at "review" in Story 3.3 — workflow sync gap

Story 3.3's sprint-status was not updated from "review" to "done" by the dev agent — it required manual correction at review. The review agent caught this, but it represents a workflow sync gap: the dev agent completed the story but failed to update the sprint-status file.

**Root cause:** The sprint-status update is the last action in the story workflow, and it was skipped when the dev agent concluded its work. The closing checklist item needs to explicitly include "Update sprint-status.yaml."

### 4. Test count discrepancies in Completion Notes — persistent LOW-grade issue

Despite P-1 improvement, 3 out of 6 stories had incorrect test counts in their Completion Notes (undercounting by 6-16 tests). Review corrected these each time. The root cause is that developers count expected tests from the task spec, not actual tests after implementation.

**Action item (P-4):** Add to closing checklist: "Run `pytest --collect-only -m 'not gpu' -q | tail -1` and use the exact count in Completion Notes." This is a single command that gives ground truth.

### 5. Stub tests broke when implementations arrived — not tracked proactively

Stories 3.5 and 3.6 both had to update existing stub tests:
- Story 3.5: `test_visualize_raises_not_implemented_error` → must be changed to assert `AdapterError`
- Story 3.6: `test_export_raises_not_implemented_error` + `test_export_error_message_references_story_36` → must be updated

This is expected behavior, but it wasn't in the dev notes as a "task" — devs had to discover it when tests failed. The Dev Notes pattern of "Files Being Modified — Current State" already shows the current `NotImplementedError` stubs, but it should explicitly say "Update these two stub tests."

**Action item (P-5):** Story Dev Notes template should include a "Stub tests to update in this story" section for stories implementing previously-stubbed methods.

## Key Insights

**Insight 1 — The deferred import convention needs a priority exception rule**

Stories 3.2–3.6 established a strong convention: all third-party imports deferred inside method bodies. This is correct for performance. But Story 3.4 revealed a conflict: when a guard clause must run before backend libraries are available, the guard MUST precede even the deferred imports. The convention needs an explicit exception: "Guard clauses (pre-flight checks) run before deferred imports. This is not an exception to the convention — it IS the convention, correctly applied."

**Insight 2 — Two-path branching (debug/non-debug) sets a precedent for Epic 4**

The `debug_hooks=True/False` branching pattern in `fit()` (Stories 3.3, 3.4, 3.5, 3.6 all maintained both paths) demonstrates that `fit()` can cleanly branch on optional features. Epic 4 will add invariant checking inside `fit()` — the same pattern applies. The precedent is established.

**Insight 3 — `_FROM_SCRATCH_BASELINE` constants are well-documented for reuse**

Story 3.5 introduced `_FROM_SCRATCH_BASELINE_SECONDS = 72.0 * 3600.0` and `_FROM_SCRATCH_BASELINE_LABEL` as module-level documented constants in `utils/visualization.py`. This is exactly the pattern for traceable benchmarks. Epic 4 compliance performance benchmarks (NFR-05: < 30 seconds for 1000 trajectories) should follow the same constant-documentation approach.

**Insight 4 — The `_share_panel()` Colab path is untested — real remote required**

`_share_panel()` uses `import google.colab` detection and `IPython.display.Javascript` for clipboard copy. The non-Colab fallback is tested. The Colab path is not tested in CI (it can't be). This is acceptable for a share feature. But the risk is that the `YOUR-ORG` URL that gets copied will be broken until D-1 is resolved. Epic 4 cannot depend on any Colab URL behavior until YOUR-ORG is configured.

**Insight 5 — 18 GPU tests written but never run — accumulating test debt**

GPU tests from Stories 3.2–3.6 (`@pytest.mark.gpu`) are syntactically correct and architecturally right, but have never been run on real T4 hardware. They test:
- VRAM < 8 GB (NFR-04)
- 10k steps < 45 min (NFR-03)
- Checkpoint write/load on actual model weights
- Visualization inference pass on GPU

If these tests contain bugs, they won't be discovered until GPU infra is available. This is a risk that grows with each deferred GPU test story.

## Significant Discoveries

**Discovery 1 — `fit()` needs callback registration for Epic 4**

The current `DreamerV3Adapter.fit()` training loop is a monolithic function with no per-step callback mechanism. Story 4.3 requires injecting invariant checks per trajectory. This means either:
- (a) Adding an `_invariants: list[...]` instance variable and checking it inside the training loop
- (b) Adding a per-step callback hook parameter to `fit()`

Option (a) is cleaner (follows the `_loss_history` pattern already established). The Story 4.3 Dev Notes should specify which approach is taken and where in the training loop the check occurs (pre-batching vs. post-step).

**Impact:** Story 4.3 Dev Notes need explicit guidance on this integration point before implementation starts. This does NOT require updating Epic 4's planned stories — but does require an alignment session (or a research spike) before Story 4.3 is written.

**Discovery 2 — `_triptych_path` and `_last_checkpoint_path` are state that persists across fit() calls**

Both `_triptych_path` (set by `visualize()`) and `_last_checkpoint_path` (set by `fit()`) are intentionally NOT reset in `_reset_training_state()`. This means:
- A second `fit()` call will update `_last_checkpoint_path` but leave `_triptych_path` pointing to the previous visualization.
- `export()` will export the OLD triptych if `visualize()` isn't called again after the second `fit()`.

This is documented behavior (the docstrings are clear) but it's a subtle state model that Epic 4 stories should be aware of when they extend `fit()`.
