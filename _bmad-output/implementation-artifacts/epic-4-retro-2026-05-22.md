### 2. _HasInvariants Protocol — elegant solution to a hard constraint

Story 4.3 faced a genuine architectural challenge: `register_invariant()` in `core/validation.py` needs to attach `_InvariantRecord` objects to a `DreamerV3Adapter` instance — but `core/ → adapters/` imports are forbidden by `test_core_boundary.py`, **which uses AST walking** (not just runtime checks). A `TYPE_CHECKING` import would have been caught.

The solution: define a `_HasInvariants` Protocol in `core/validation.py` that duck-types the adapter's `_invariants` attribute. No import from `adapters/` needed — the protocol is defined in `core/`, the adapter satisfies it at runtime without any special declaration.

Alice (Product Owner): "This is the pattern Epic 6 will need when Samuel's domain scientist persona interacts with the adapter. We have a reusable precedent."

Dana (QA Engineer): "And it's completely transparent to the user — they just call `register_invariant(adapter, ...)`. They never see the Protocol."

### 3. NFR-05 massively beaten: 0.55ms vs 30s limit

The compliance report computation on 1000 trajectories was benchmarked at approximately 0.55 milliseconds mean execution time — achieving a 54,000× margin against the 30-second NFR-05 limit. This is not a fluke; it reflects the architecture decision to make `ComplianceReport` a pure data object with no computational overhead in `compliance_report()` (just list arithmetic over stored residuals).

Charlie (Senior Dev): "We over-engineered the performance requirement in the best possible way. The 30s limit was conservative for pure Python list arithmetic."

## Challenges & Growth Areas

### 1. Type annotation precision — recurring MEDIUM pattern across 3 stories

Stories 4.3, 4.4, and 4.5 all had type annotation issues caught at review or in completion notes:
- Story 4.3: review corrected `tolerance < 0` → `ConfigurationError` (not `ValidationError` as originally documented)
- Story 4.4: post-review fix corrected `list[dict]` → `list[dict[str, Any]]` in `compliance_report()` (mypy strict `[type-arg]` rule)
- Story 4.5: `tmp_path: pytest.MonkeyPatch` → `tmp_path: Path` (MEDIUM finding at review)

**Root cause:** The `dict[str, Any]` requirement is not in the "common traps" reference material. Developers writing `list[dict]` is natural Python but fails mypy strict `[type-arg]`. The rule is non-obvious.

**Action item (P-7):** Add to shared "Common Python Traps" note: "mypy `--strict` requires fully typed generics: use `dict[str, Any]` not bare `dict` when dict values are heterogeneous. Same for `list[Any]` vs `list`. Bare generics trigger `[type-arg]` errors."

### 2. test fixture type annotation error repeated: `pytest.MonkeyPatch` vs `Path`

Story 4.5 had a MEDIUM finding: `tmp_path` typed as `pytest.MonkeyPatch` instead of `pathlib.Path`. This is a conftest.py fixture where `tmp_path` is always `pathlib.Path`. The error is subtle because both types are imported from `pytest` in typical test files.

**Root cause:** Developer copied a test structure and didn't update the type annotation for `tmp_path`.

**Action item (P-8):** Add to Dev Notes template test section: "`tmp_path` fixture is `pathlib.Path`, NOT `pytest.MonkeyPatch`. Never annotate `tmp_path` as `MonkeyPatch`."

### 3. AST boundary test stricter than documented — catches TYPE_CHECKING imports

Story 4.3 debug log documents a critical discovery: `test_core_boundary.py` uses AST walking and catches `from physlink.adapters.dreamer import DreamerV3Adapter` even inside a `TYPE_CHECKING` block in a `core/` module. The documentation in Epic 3's Dev Notes warned about the boundary but did not explicitly state that `TYPE_CHECKING`-guarded imports are also caught.

This led the dev agent to discover and implement the `_HasInvariants` Protocol approach — which is architecturally superior but was not the originally planned approach.

**Action item (P-9):** Update architecture.md boundary documentation: "The `test_core_boundary.py` AST walk catches ALL imports from `adapters/` in `core/` files — including imports under `if TYPE_CHECKING:`. Use Protocol (defined in `core/`) to type-hint adapter parameters in `core/` modules. This is the established pattern."

### 4. Deprecated PyTorch API: `torch.cuda.amp.autocast` → `torch.amp.*`

Story 4.3 review caught usage of `torch.cuda.amp.autocast` and `torch.cuda.amp.GradScaler` (deprecated in PyTorch 2.x in favor of `torch.amp.autocast` and `torch.amp.GradScaler`). These were introduced in Stories 3.x and persisted until Story 4.3 review.

**Root cause:** The "Common Python Traps" reference was not extended to cover PyTorch API deprecations.

**Action item (P-10):** Add to shared "Common Python Traps": "Use `torch.amp.autocast(device_type='cuda')` and `torch.amp.GradScaler()` — the `torch.cuda.amp.*` namespace is deprecated in PyTorch 2.x."

## Key Insights

**Insight 1 — Protocol pattern for cross-boundary type hints is established and reusable**

The `_HasInvariants` Protocol in `core/validation.py` is the canonical approach for when `core/` code needs to type-hint an `adapters/` object. This pattern is generalizable: any future `core/` module that needs to operate on an adapter object should define a Protocol with the minimum interface required. Epic 6 will likely need this for the domain scientist notebook interaction.

**Insight 2 — Pure data objects scale cleanly: 4 stories built one class in layers**

`ComplianceReport` was built across 4 stories:
- Story 4.3: storage (`_invariant_residuals` on adapter)
- Story 4.4: access (`summary()`, `violations()`)
- Story 4.5: visualization (`plot()`), persistence (`export()`), full residuals

At no point did a later story break the earlier implementation. The key: keeping `ComplianceReport` as a pure data object (no side effects, no adapter reference) allowed each story to extend it without touching previous tests. This is the architecture pattern that enables safe incremental delivery.

**Insight 3 — NFR margin is diagnostic information**

NFR-05 (30 seconds for 1000 trajectories) was met with 0.55ms — a 54,000× margin. This is valuable diagnostic information, not just a green checkbox. It means:
- The compliance computation is O(n) list arithmetic — any future regression to a slower algorithm would still have enormous headroom
- The 30-second limit was set for CPU-only CI (GitHub Actions slow path); real users on T4 get even more headroom
- If compliance check performance degrades in future epics (e.g., complex invariant functions), the benchmark will catch it before it becomes a user problem

**Insight 4 — The GPU test debt is accumulating with each epic**

Epic 3 left 18 GPU tests unrun. Epic 4 has added an unknown number of GPU-related tests (18 deselected throughout the epic — likely ~25+ total now including NFR-05 GPU path). These tests have never run on T4 hardware. They cover:
- VRAM < 8 GB (NFR-04)

- Invariant checking during real training

If the T4 runner is not set up before Epic 6, the project will ship with 25+ tests that have never been validated on the target runtime.

**Insight 5 — The 7-symbol API is the first deliverable Petra sees**

The API stability gate (`test_story45_final_api_symbols`) is now exact equality. This is Petra's confidence signal: the public surface is stable, contractual, and does not change accidentally. Combined with the CHANGELOG (Epic 5, already done) and GitHub templates (Epic 5), the institutional trust package is essentially complete. Epic 6 is the last mile for the domain scientist.
