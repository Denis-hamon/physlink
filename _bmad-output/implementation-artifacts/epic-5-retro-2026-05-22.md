## Challenges & Growth Areas

### 1. Dev Notes "No Tests Required" guidance was incorrect — and resulted in MEDIUM review issue

Story 5.1's Dev Notes explicitly stated "No Tests Required" but the dev agent correctly created 34 integration tests. This contradiction was caught as a MEDIUM issue in the review ("beneficial contradiction — tests improve confidence in AC validation"). The risk: future dev agents on similar stories might follow the explicit "no tests" guidance and skip tests entirely.

**Root cause:** The "No Tests Required" instruction reflected the planning intent (no pytest unit tests for Python source), not the appropriate implementation behavior (integration tests for document content validation are valuable and correct).

**Action item (P-11):** Update the Epic 5 content-story Dev Notes template to read: "No Python unit tests required. Integration tests validating document content (existence, keyword presence, format checks) are **expected and required** — follow the `tests/integration/test_changelog_content.py` pattern."

### 2. File List incompleteness — repeated pattern from Epic 4

Story 5.1's file list omitted `src/physlink/__init__.py` (modified for version bump) and `tests/integration/test_changelog_content.py` (created during implementation). This was caught at review and fixed. The same pattern (incomplete file list at story completion) was noted in Epic 4 retrospective.

**Root cause:** The dev agent completed implementation, then wrote the file list from memory rather than from `git diff --name-only`. The story template doesn't enforce "run git diff to populate file list."

**Action item (P-12):** Add to Dev Notes file list section template: "**Before marking done**, run `git diff --name-only HEAD~1 HEAD` to get the authoritative changed-files list. Never write the file list from memory."

### 3. Story 5.3 uncommitted test extensions

**Root cause:** The developer extended tests after verifying they pass, but didn't create a follow-up commit for the extended test file.

**Action item (P-13):** The 17 uncommitted tests in `tests/integration/test_github_templates.py` must be committed before Epic 6 Story 6.1 starts. Add to "Definition of Done": "All locally-created test files must be committed. Check with `git status` before marking a story done."

## Key Insights

**Insight 1 — Documentation testing is first-class quality enforcement**

The 86 integration tests added in Epic 5 are not "extra" — they are the CI enforcement layer for Petra's institutional evaluation. If any test fails:
- `test_changelog_has_three_releases` fails → Petra's hard NO-GO
- `test_no_adaptation_job_in_code_blocks` fails → import error in her copy-paste
- `test_pr_template_contains_changelog_checkbox` fails → contributor process breaks

This pattern should be the default for any documentation-heavy story: create integration tests that validate the documentation content that matters to the user persona.

**Insight 2 — "Copy-pasteable" must be enforced by testing, not by review alone**

Story 5.2's HIGH issue (undefined `trajectories`) would have passed static reading but fails when executed. The review correctly caught it, but the lesson is: code examples in documentation should be tested for executability when possible. For Epic 6 (Colab notebook), Cell 1 through Cell 8 should be validated in CI using nbconvert or a notebook execution test.

**Insight 3 — The institutional trust package is complete for Petra**

With Epic 5 done, Petra's evaluation checklist is fully addressable:
- ✅ CHANGELOG.md with 3+ dated releases (hard NO-GO resolved)
- ✅ MIT license (pre-existing)
- ✅ Lab Adoption Guide with BibTeX, named-run examples, buffer persistence
- ✅ GitHub contribution templates including domain extension path
- ⚠️ YOUR-ORG placeholders mean some links (Colab, docs badge) won't resolve until D-1 is fixed

The remaining blocker is infrastructure (D-1), not content.

**Insight 4 — Test count tracking is now reliable**

After the MEDIUM issues in Epic 4 (test count discrepancies), Epic 5 shows improvement: Story 5.2 explicitly documented "27 tests / 9 classes" before marking done, and the review only found 1 discrepancy (redundant test removed). The `pytest --collect-only` pattern from Epic 4's P-4 action item is holding.

**Insight 5 — Content stories benefit from the same review discipline as code stories**

All 3 Epic 5 stories had genuine review findings (HIGH: 1, MEDIUM: 7, LOW: 2). The assumption that "content-only = lower review burden" is incorrect. Story 5.2 had a HIGH issue (NameError) that would have directly failed Petra's evaluation. Content quality gates are as important as code quality gates.
