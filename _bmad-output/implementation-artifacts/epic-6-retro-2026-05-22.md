## Challenges & Growth Areas

### 1. Pre-Epic-4 placeholder content in docs/domain-scientists.md was invisible risk

The most significant challenge in Epic 6 was discovered in Story 6.2: `docs/domain-scientists.md` was created as a "DD-003: Samuel" placeholder during early planning and never updated after Epic 4 implemented the actual `register_invariant()` API. The placeholder had:
- Wrong import: `from physlink.compliance import register_invariant` (doesn't exist)
- Wrong API: `register_invariant(space=obs_space, name=..., fn=..., severity=...)` (wrong signature)
- Wrong return type: `fn` returning a bool (should return float residual)
- Wrong output format: emoji table (should be `name: PASS (max_residual=X, threshold=Y, violations=Z/N)`)

**Root cause:** Planning artifacts (doc placeholders) created before the implementation API was finalized became stale without any marker indicating staleness. The dev agent for Story 6.2 correctly identified this and rewrote from scratch, but any review or QA step that checked the file between Epic 4 and Epic 6 would have found incorrect API documentation.

**Action item (P-14):** Pre-Epic placeholder doc files should include a comment: `<!-- PLACEHOLDER: pre-Epic-X content. Must be fully rewritten by Story Y.Z. Do not use as API reference. -->`. This creates a visible flag that prevents accidental use of placeholder content.

### 2. Colab notebook is validated structurally but not executed in CI

Epic 5 Insight 2 anticipated this: "For Epic 6 Colab notebook, Cell 1 through Cell 8 should be validated in CI using nbconvert or a notebook execution test." Story 6.3 addressed this with 58 structural/static tests (cell count, cell type, source content inspection) but NOT with actual execution tests.

The 58 tests verify:
- Correct JSON structure ✅
- Correct cell count and types ✅
- Correct source content (subprocess guard, ⚠️ marker, register_invariant call, etc.) ✅

They do NOT verify:
- That `DreamerV3Adapter.fit()` actually runs on the given trajectory shape ❌
- That `report.summary()` produces the expected PASS string ❌
- That `report.plot()` renders without matplotlib errors ❌

**Root cause:** D-2 (GPU runner / T4 infra) is not resolved. Without a CI runner with GPU support (or at least CPU execution capability for a 100-step fit), nbconvert execution tests would be flaky or require special infrastructure.

**Action item (P-15):** Add a `tests/integration/test_notebook_execution.py` that runs `nbconvert --to notebook --execute` on `notebooks/domain-scientist-colab.ipynb` in a headless CPU mode with `steps=10` override. This validates executability without requiring T4. Mark with `@pytest.mark.notebook_execution` and run separately from the main suite.

D-1 was first flagged in Epic 1. It was elevated to "BLOCKER for Epic 6" in Epic 5 retro. Epic 6 proceeded anyway — correctly, because the content stories could use `YOUR-ORG` as a consistent placeholder and deliver full value. The find-and-replace + git remote configuration is a configuration task, not a development task. It unblocks every user-facing URL in the project."

### 4. Test count documentation MEDIUM pattern persisted into Epic 6

Epic 4 and 5 retroed on test count documentation inconsistencies. Epic 6 had 3 MEDIUM review findings — all test count corrections:
- Story 6.1: documented 5 tests, created 8 (QA auto-gap caught this at review)
- Story 6.3: documented 46 tests in initial notes, created 58 (corrected to 58 at review)
- Story 6.3: documented 894 passed, actual 907 (corrected at review)

The process improvement from P-4 (Epic 4) — "use `pytest --collect-only` before writing test count" — is not consistently applied. The QA-gap analysis workflow adds tests after the initial count is written, creating a timing problem.

**Root cause:** The test count in Dev Notes is written before the QA-gap analysis is run. QA-gap analysis adds tests, making the initial count stale.

**Action item (P-16):** Test count in Completion Notes should be written AFTER all QA-gap tests are added, not before. Sequence: (1) implement core tests → (2) run QA-gap analysis → (3) add QA-gap tests → (4) run `pytest --collect-only` → (5) write Completion Notes with final count.

## Key Insights

**Insight 1 — Stale placeholder content is invisible debt**

`docs/domain-scientists.md` was functionally broken for 2+ epics (Epics 5 and 6 relied on it being correct, but it had been a pre-Epic-4 placeholder the whole time). The content was wrong but syntactically valid — no linter would catch it. The only detection mechanism was a developer reading the Epic 4 architecture carefully before implementing Story 6.2.

This is a structural risk for any project where doc files are created as planning artifacts before the implementation API is finalized. The correct mitigation is a marker protocol (P-14 action item).

**Insight 4 — QA-gap analysis generates lasting value beyond the story**
