## Overview

**Page Purpose:** Let Hugo define a 7-DOF robot through a short, recognizable API and verify that PhysLink understands the observation and action spaces before adaptation begins.

**User Situation:** Hugo has passed `physlink doctor` and is ready to test whether PhysLink can represent his domain without hiding critical assumptions.

**Main User Goal:** Configure `ObservationSpace` and `ActionSpace` in fewer than 15 lines of domain code, then inspect the structured summary and explanation.

**Success Criteria:** Hugo sees validated observation/action dimensions, understands normalization/clipping behavior, and continues to the DreamerV3 adaptation loop.

**Entry Points:**
- `Continue to Universal Space API configuration` from `1.2-colab-setup`.
- Direct anchor link to the notebook configuration section.

**Exit Points:**
- Primary: continue to `1.4-colab-adaptation-loop`.
- Secondary: open API reference or troubleshooting if validation fails.
- Tertiary: return to setup diagnostics.

## Layout Structure

The configuration section should feel like a lab notebook checkpoint: define the domain, run a validation, inspect the exact assumptions, then move forward.

```text
+----------------------------------------------------------+
| Configuration Header                                     |
| section title, moment-of-truth framing, input promise     |
+----------------------------------------------------------+
| Domain Preset Context                                    |
| 7-DOF robot preset and editable assumptions               |
+----------------------------------------------------------+
| Observation Space Cell                                   |
| ObservationSpace.from_proprioception(...)                 |
+----------------------------------------------------------+
| Action Space Cell                                        |
| ActionSpace.continuous(...)                              |
+----------------------------------------------------------+
| Space Summary Cell                                       |
| obs.summary(), act.summary(), dimension checks            |
+----------------------------------------------------------+
| Explainability Panel                                     |
| obs.explain(), act.explain(), transforms and constraints  |
+----------------------------------------------------------+
| Validation Feedback                                      |
| success, warnings, dimension/type errors                  |
+----------------------------------------------------------+
| Next Step Gate                                           |
| continue to adaptation loop or fix configuration          |
+----------------------------------------------------------+
```

## Page Interactions

| Object ID | Interaction |
|-----------|-------------|
| `colab-config-observation-code-cell` | User runs or edits the cell. PhysLink validates `joints` and `include_velocity`, then creates `obs`. |
| `colab-config-action-code-cell` | User runs or edits the cell. PhysLink validates `dims` and `bounds`, then creates `act`. |
| `colab-config-summary-code-cell` | User runs the cell. The notebook displays dimensions, bounds, dtypes, and adapter compatibility. |
| `colab-config-explain-code-cell` | User runs the cell. The notebook displays transformation metadata for observation and action spaces. |
| `colab-config-next-adaptation-link` | Moves to the adaptation loop only after the valid configuration state. |
| `colab-config-next-api-reference-link` | Opens Universal Space API reference in a new tab. |

**Global Interaction Rules:**
- Configuration code must remain under 15 domain lines excluding imports.
- Validation should run as early as possible, ideally during object construction.
- Error messages must name the failing parameter, the received value, and the expected correction.
- The notebook must avoid hiding transformation behavior behind prose; `.explain()` is the trust surface.

## Validation & Errors

| Field / Target | Rule | Timing | Error / Recovery |
|----------------|------|--------|------------------|
| `joints` | Must be a positive integer. | `ObservationSpace.from_proprioception(...)`. | `Invalid joints: expected positive integer, received {value}.` |
| `include_velocity` | Must be boolean. | `ObservationSpace.from_proprioception(...)`. | `Invalid include_velocity: expected True or False.` |
| Observation dims | With `joints=7` and velocity enabled, inferred dims should be 14. | Observation construction / summary. | `Observation dimension check failed. Verify joints and velocity settings.` |
| `dims` | Must be a positive integer and match robot action dimension. | `ActionSpace.continuous(...)`. | `Action dimension mismatch: expected dims=7 for this robot preset.` |
| `bounds` | Must be numeric `(min, max)` tuple with `min < max`. | `ActionSpace.continuous(...)`. | `Invalid bounds: provide a numeric (min, max) tuple such as (-1.0, 1.0).` |
| Summary compatibility | Observation and action spaces must be adapter-compatible. | `summary()` call. | `Space summary failed: {reason}. Update the highlighted configuration value.` |
| Explainability availability | `.explain()` must return transformation metadata. | `explain()` call. | `Explainability metadata unavailable. Rerun configuration cells before continuing.` |
| Domain code length | Domain configuration should stay under 15 lines excluding imports. | Notebook review / docs test. | `Configuration example exceeds 15 lines; simplify before publication.` |

## Conditional Sections

| Condition | Include |
|-----------|---------|
| Public notebook | Keep all example values editable and explain what can be changed. |
| Needs API data | No network API required; all validation is local. |
| Error recovery | Every invalid parameter needs exact expected type/value guidance. |
| Accessibility | Do not rely on green/red color only; use `[OK]`, `[WARN]`, `[FAIL]` labels. |
| Open questions | Confirm final API signatures and whether `.summary()` lives on spaces, adapter, or both. |
