## Overview

**Page Purpose:** Start the DreamerV3 adaptation run and keep Hugo oriented while the notebook performs a long-running model operation.

**User Situation:** Hugo has valid `ObservationSpace` and `ActionSpace` objects. He is ready to find out whether PhysLink can actually adapt a World Model without opaque failure.

**Main User Goal:** Initialize `DreamerV3Adapter`, run `.fit(trajectories, steps=10_000)`, and monitor progress without guessing whether the run is healthy.

**Success Criteria:** Hugo sees a credible progress indicator, ETA, prediction health status, checkpoint activity, and a clear route to validation.

**Entry Points:**
- `Continue to DreamerV3 adaptation` from `1.3-colab-configuration`.
- Direct anchor link to the adaptation section.

**Exit Points:**
- Primary: continue to `1.5-colab-validation`.
- Secondary: inspect debug hooks if prediction health becomes `ANOMALY`.
- Tertiary: recover from checkpoint after disconnect or runtime interruption.

## Layout Structure

The adaptation loop should feel like a flight instrument panel inside the notebook: one command starts the run, then the page explains whether the system is progressing, drifting, checkpointing, or needs attention.

```text
+----------------------------------------------------------+
| Adaptation Header                                        |
| section title, run promise, expected duration             |
+----------------------------------------------------------+
| Adapter Initialization Cell                              |
| DreamerV3Adapter(obs, act)                               |
+----------------------------------------------------------+
| Trajectory Source Cell                                   |
| load or synthesize quickstart trajectories                |
+----------------------------------------------------------+
| Fit Cell                                                 |
| adapter.fit(trajectories, steps=10_000)                  |
+----------------------------------------------------------+
| Progress Monitor                                         |
| progress bar, ETA, step count, prediction health          |
+----------------------------------------------------------+
| Debug Hooks Panel                                        |
| inspect prediction anomalies and pipeline stages          |
+----------------------------------------------------------+
| Checkpoint Panel                                         |
| last checkpoint, export path, recovery state              |
+----------------------------------------------------------+
| Next Step Gate                                           |
| continue to validation or inspect debug details           |
+----------------------------------------------------------+
```

## Spacing

**Scale:** WDS local tokens: `zero`, `sm`, `md`, `lg`, `xl`, `2xl`, `3xl`.

| Property | Token |
|----------|-------|
| Notebook section padding | Colab markdown default |
| Section gap | `lg` default |
| Code cell to output gap | `sm` |
| Monitor row gap | `sm` |
| Major state transition gap | `xl` |

#### <-> `colab-loop-v-space-sm` - header and initialization stay connected

The user should move quickly from explanation to adapter initialization.

#### <-> `colab-loop-v-space-md-trajectories` - adapter to data source

Trajectory source deserves its own boundary because it affects trust in the run.

#### <-> `colab-loop-v-space-sm-fit` - data source and fit command stay coupled

The fit cell should feel like the direct next action after trajectory loading.

#### <-> `colab-loop-v-space-md-monitor` - fit command to monitoring panel

The monitoring panel is the result surface for the long-running action.

#### <-> `colab-loop-v-space-lg-debug` - monitor to debug hooks

Debug hooks are secondary until an anomaly appears, but must be close enough to inspect.

#### <-> `colab-loop-v-space-lg-checkpoint` - debug hooks to checkpoint recovery

Checkpointing is a separate confidence layer for Colab interruption risk.

#### <-> `colab-loop-v-space-lg-next` - checkpoint panel to next gate

The next action should appear after run health and recovery state are understood.

| Object ID | Interaction |
|-----------|-------------|
| `colab-loop-adapter-code-cell` | User runs the cell. Adapter initializes from validated `obs` and `act`; summary confirms compatibility. |
| `colab-loop-trajectories-code-cell` | User runs the cell. Trajectory data loads and summary checks shape compatibility. |

| `colab-loop-checkpoint-export-cell` | User runs the cell to export YAML with spaces and checkpoint path. |

**Global Interaction Rules:**
- Long-running output must update at least every 30 seconds while active.
- Prediction status must expose both normal and anomaly states.
- Checkpoint state must be visible without requiring source-code inspection.
- Recovery instructions must survive a Colab reconnect.

## Page States

| State | When | Appearance | Actions |
|-------|------|------------|---------|
| Default | Page opens after valid configuration. | Header, adapter cell, trajectory cell, fit cell, and empty monitor are visible. | Run adapter cell. |
| Adapter ready | Adapter initializes successfully. | `[OK] DreamerV3Adapter initialized...` output appears. | Load trajectories. |
| Trajectories ready | Trajectory summary matches configured spaces. | Summary table shows matching observation/action dims. | Run fit cell. |
| Fit running | `.fit(...)` is active. | Progress bar, ETA, throughput, prediction health, and checkpoint status update. | Monitor or open debug hooks. |
| Prediction OK | Model health is within expected thresholds. | `Prediction: OK` status badge. | Continue monitoring. |
| Prediction anomaly | Prediction health crosses anomaly threshold. | `Prediction: ANOMALY`; debug panel is emphasized. | Inspect debug hooks. |
| Checkpoint saved | Checkpoint policy triggers. | Latest checkpoint row updates with step and path. | Continue or export. |
| Colab disconnected | Runtime disconnects during fit. | Native Colab disconnect plus recovery note. | Reconnect and resume from checkpoint. |
| Fit complete | Requested steps complete. | Progress shows 10,000/10,000 and result summary appears. | Export and continue to validation. |
| Fit failed | Training raises an unrecoverable error. | Error callout names failure stage and latest checkpoint. | Inspect debug hooks or resume/retry. |
| Export complete | `adapter.export(...)` succeeds. | YAML path is shown. | Continue to validation. |

## Validation & Errors

| Field / Target | Rule | Timing | Error / Recovery |
|----------------|------|--------|------------------|
| Adapter initialization | Must receive valid `ObservationSpace` and `ActionSpace`. | `DreamerV3Adapter(obs, act)`. | `Adapter initialization failed: rerun configuration cells and summary checks.` |
| Trajectory shape | Observation/action dimensions must match configured spaces. | Trajectory load / summary. | `Trajectory shape mismatch: expected obs={obs_dims}, act={act_dims}.` |
| Fit steps | `steps` must be a positive integer; quickstart target is 10,000. | `.fit(...)` call. | `Invalid steps: provide a positive integer such as 10000.` |
| Progress updates | Long-running output should update regularly. | During `.fit(...)`. | `Progress stalled. Inspect runtime status and latest checkpoint.` |
| Prediction health | Must report `OK` or `ANOMALY`; unavailable status is an error for the quickstart. | During `.fit(...)`. | `Prediction health unavailable. Open debug hooks before continuing.` |
| Checkpoint save | Must save at least every 10 minutes or 1,000 steps. | During `.fit(...)`. | `Checkpoint save failed. Export current run state or reduce checkpoint path permissions.` |
| Colab recovery | Latest checkpoint must be usable after reconnect. | After interruption. | `No usable checkpoint found. Restart from the last completed setup cell.` |
| Export YAML | `adapter.export()` must produce valid YAML with spaces and checkpoint path. | Export cell. | `Export failed or YAML invalid. Rerun export after a successful checkpoint.` |

## Conditional Sections

| Condition | Include |
|-----------|---------|
| Public notebook | Keep quickstart trajectories reproducible and clearly labeled. |
| Long-running task | Provide ETA, progress, throughput, prediction health, and checkpoint state. |
| Debug needed | Surface debug hooks on anomaly without forcing advanced users into source code. |
| Colab interruption | Recovery path must be visible and tied to latest checkpoint. |
| Accessibility | Do not rely on animated progress alone; expose text status and step counts. |
| Open questions | Confirm final debug hook API, checkpoint interval policy, and export YAML schema. |
