## Overview

**Page Purpose:** Let Petra verify maintenance cadence and release regularity in under 2 minutes. The CHANGELOG is not a marketing page — it is a technical accountability document. Its quality directly signals the project's long-term viability.

**User Situation:** Petra has just left the GitHub README (2.1) with her initial GO assessment based on badges. She is now on the CHANGELOG to answer a more precise question: *"Is this project actually maintained, or was it launched and then quietly abandoned?"*

**Main User Goal:** Confirm release regularity (cadence), release completeness (semantic versioning + human-readable descriptions), and absence of reckless breaking changes before proceeding to the ArXiv paper.

**Success Criteria:** Petra sees at least 3 releases with meaningful descriptions, a consistent cadence (no gap > 6 months), and no undocumented breaking changes. She forms her GO decision and navigates to the ArXiv paper (2.3).

**Entry Points:**
- Click on `Changelog` link from `readme-links-changelog-link` (2.1 Deep Trust Links section).
- Click on the `readme-repository-maintenance-badge` (2.1 Repository Header) if it links to the Releases page.

**Exit Points:**
- Primary: Navigate to ArXiv paper (2.3) — via remembered arXiv badge or back to README → "Read the paper."
- Secondary: Navigate back to GitHub README (2.1) to recollect the ArXiv link.
- NO-GO exit: Close the tab entirely. PhysLink is not recommended.

## Petra's Reading Pattern

Petra reads the CHANGELOG in a specific sequence that differs from a developer's debug-oriented read. She is not looking for what changed technically — she is looking for signals of institutional reliability.

| Step | Petra's action | Signal she's extracting |
|------|----------------|------------------------|
| 1 | Scans release entry headers top to bottom (`## [1.2.0] - 2026-03-15`) | Counts releases; reads dates to assess cadence |
| 2 | Checks the most recent release date | Is the project active? Last release < 6 months = GO |
| 3 | Counts total releases | Is there a history? ≥ 3 releases = provisional GO |
| 4 | Reads the first 2 release entries (Added / Changed / Fixed) | Are descriptions human-readable? Or just commit hashes? |
| 5 | Scans for `Breaking Changes` or `⚠️` markers | Are breaks documented with migration paths? |
| 6 | Notes the [Unreleased] section if present | Is active development underway? |

**GO condition:** Recent release (< 6 months), ≥ 3 total releases, human-readable descriptions, no undocumented breaking changes.

**NO-GO conditions (any one is sufficient):**
- Last release > 12 months ago with no Unreleased section → abandoned.
- Only 1 release (v0.1.0 or v1.0.0 only) → just launched, no track record.
- Entries are commit hashes or vague ("various fixes") → poor project hygiene.
- Breaking changes with no migration guidance → reckless project management.

## Spacing

**Scale:** WDS local tokens: `zero`, `sm`, `md`, `lg`, `xl`, `2xl`, `3xl`.

> GitHub controls all spacing in the file view wrapper. For CHANGELOG content, spacing is controlled by Markdown heading and list conventions — no custom CSS applies.

| Property | Convention |
|----------|------------|
| Between release entries | Blank line before each `##` heading (renders as visual separation in GitHub Markdown) |
| Between category sections | Blank line before each `###` heading |
| Between bullet items | Standard Markdown list (tight — no blank lines between items) |
| Breaking change callout | Visually separated from normal items — use `⚠️` emoji prefix and a blank line before the migration note |

#### ↕ `changelog-v-space-release-entry` — each release entry is a discrete unit

Blank lines before `## [version]` headings create GitHub's native vertical rhythm between entries. Petra reads these visually as separate chapters.

#### ↕ `changelog-v-space-category` — category headers group related changes

`### Added`, `### Changed`, `### Fixed` act as sub-sections within a release entry. Tight spacing between bullet items; breathing room before and after each category heading.

#### ↕ `changelog-v-space-breaking` — breaking changes require visual isolation

A breaking change item must be preceded by a blank line and a `⚠️` marker to catch Petra's eye during her fast scan. Migration note follows immediately below the breaking change item.

## Typography

GitHub controls the rendered typography. CHANGELOG.md content uses standard Markdown heading levels. Semantic use of heading levels ensures correct rendering in GitHub's Markdown stylesheet.

| Element | Markdown | GitHub renders as | Role |
|---------|----------|-------------------|------|
| Page title | `# Changelog` | H1 — large, bold | File section header |
| Release entry header | `## [1.3.0] - 2026-04-10` | H2 — prominent | Petra's primary scan target |
| Category header | `### Added` | H3 — medium | Petra reads to assess content quality |
| Bullet item | `- description` | Unstyled list item | The actual change description |
| Breaking change warning | `⚠️ **Breaking:** description` | Bold within list item | High-visibility marker Petra scans for |
| Migration note | `> **Migration:** explanation` | Blockquote | Immediately follows a breaking change item |
| Unreleased header | `## [Unreleased]` | H2 — same as release | Signals active work in progress |

**Typography rule:** Release entry headers (`## [version] - date`) are the most important typographic element on this page for Petra. They carry both the version number (history depth) and the date (cadence). The date must always be included — a release header without a date is a quality failure.

## Validation & Errors

Validation for CHANGELOG.md is entirely pre-release / documentation quality review. No runtime validation applies.

| Field / Target | Rule | Timing | Error / Recovery |
|----------------|------|--------|------------------|
| Release header format | Must follow `## [MAJOR.MINOR.PATCH] - YYYY-MM-DD`. Date is required. | Pre-release CI lint (`changelog-lint` or equivalent). | `Release header missing date` — adds a date during release PR review. |
| Release content | Must contain human-readable descriptions, not commit hashes. | Pre-release review. | `CHANGELOG entry reads as commit dump` — rewrite to researcher-facing language before release. |
| Breaking change | Must be prefixed with `⚠️ **Breaking:**` and followed by a `> **Migration:**` blockquote. | Pre-release CI lint. | `Breaking change lacks migration note` — blocks release merge until migration note is added. |
| Keep a Changelog format | Should declare format compliance in the header note. | Project setup (one-time). | Missing format note reduces quality signal but does not block. |
| Version diff links | Every `## [version]` should have a corresponding compare URL at the file bottom. | Pre-release CI. | Missing compare links reduce navigability but are not blocking. |
| Release cadence | No hard rule; target: at least one release every 3-6 months while the project is active. | Qualitative — no CI check. | Cadence gaps should be documented in the Unreleased section with a status note. |

## Conditional Sections

| Condition | Include |
|-----------|---------|
| Active development | Include `## [Unreleased]` section and keep it updated. |
| Breaking change | Always include `⚠️ **Breaking:**` prefix and `> **Migration:**` blockquote. |
| First release | Minimal history is acceptable; note the launch date prominently. |
| Long-term project | Aim for ≥ 5 entries to demonstrate track record. Consider archiving very old entries (pre-v0.5.0) in a separate CHANGELOG-archive.md. |
| Multiple significant contributors | The quality and consistency of CHANGELOG writing is a community health signal — assign CHANGELOG review as a step in the PR template. |
