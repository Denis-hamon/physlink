## Overview

**Page Purpose:** This is Petra's **MOMENT OF TRUTH**. She has confirmed maintenance (README + CHANGELOG). Now she must answer Paula's likely question: *"Is there a peer-reviewed foundation behind this tool?"* The arXiv paper is PhysLink's scientific authority claim.

**User Situation:** Petra arrives from the CHANGELOG with a strong GO inclination. She is not reading the paper cover-to-cover — she is performing a 5-minute credibility scan to extract three things: (1) the paper is real and archived, (2) the authors have credible affiliations, (3) the results are quantified and honest. She then copies the citation into her memo for Paula.

**Main User Goal:** Confirm scientific depth and extract the arXiv citation for her recommendation memo, then proceed to the Lab Adoption Guide for implementation content.

**Success Criteria:** Petra reads the abstract, checks author affiliations, scans the benchmark table, finds the "Cite as" block, and copies the BibTeX or citation string. She exits to the Lab Adoption Guide (2.4) with the citation saved.

**Entry Points:**
- Click on `Read the paper` from `readme-action-read-paper-link` (2.1 Fast Action Bar).
- Click on `readme-repository-arxiv-badge` (2.1 Repository Header).
- Direct arXiv URL retained from her README session.

**Exit Points:**
- Primary: Navigate to Lab Adoption Guide (2.4) — back to README → `Lab Adoption Guide` link.
- Secondary: Download PDF for offline reading / sharing with Paula.
- Tertiary: Navigate to GitHub repository via the paper's GitHub link (Samuel's primary exit, Petra's fallback).

## Dual-Persona Scan Map

The same arXiv page serves two completely different reading strategies.

| Dimension | Petra (2.3) | Samuel (3.1) |
|-----------|-------------|--------------|
| Entry state | GO already established (README + CHANGELOG passed) | Cold discovery — neutral, skeptical |
| Primary question | "Is this academically citable for Paula?" | "Do these authors understand physics, not just ML?" |
| First object read | Author affiliations | Abstract keywords |
| Critical scan targets | Affiliations → abstract → benchmarks → citation block | Abstract (`conservation laws`, `physical invariants`) → author backgrounds → benchmark (long-horizon consistency) |
| Exit | Lab Adoption Guide (2.4) with citation copied | GitHub README (2.1) via paper's GitHub link |
| GO condition | Real paper, credible affiliations, quantified results, citable | Abstract uses physics vocabulary correctly; benchmarks show physical consistency metrics |
| NO-GO condition | Pure ML preprint with no physics rigor claims; authors from single industry lab only | Authors speak only ML language; no mention of conservation laws or domain constraints |
| Time on page | ~5 minutes (credential check + citation copy) | ~10–15 minutes (deep read of abstract, methods, invariants section) |

#### Author Affiliations

**OBJECT ID:** `arxiv-paper-affiliations`

| Property | Value |
|----------|-------|

### Section: Abstract

**OBJECT ID:** `arxiv-paper-abstract`

| Property | Value |
|----------|-------|
| Purpose | The densest signal surface on the page. Both Petra and Samuel read this fully and extract their primary GO/NO-GO signals here. |
| Priority | **PRIMARY for both personas** |
| Controlled by | Paper content (PhysLink controls). |
| Length | 150–250 words. |

#### Abstract Content Requirements

**OBJECT ID:** `arxiv-abstract-content`

| Property | Value |
|----------|-------|
| Component | Body text (arXiv-rendered) |
| Content requirement (Petra) | Must state: (1) the problem (adaptation bottleneck), (2) the approach (standardized API), (3) quantified results (e.g., "reduces adaptation time from X to Y"), (4) honest scope (e.g., "evaluated on 3 robotic domains"). |
| Content requirement (Samuel) | Must include at minimum two of: `conservation laws`, `physical invariants`, `physical constraints`, `domain-specific validation`. Absence of physics vocabulary = Samuel's NO-GO. |
| Failure mode (Petra) | Abstract full of benchmark numbers but no clear problem statement → reads as an engineering benchmark paper, not a scientific contribution. |
| Failure mode (both) | Vague claims ("achieves state-of-the-art") without numerical grounding → reduces confidence. |

**Abstract example structure (required elements):**
```
[Problem hook — 2 sentences]
[Approach — 2 sentences, names the key technical contribution]
[Results — 2 sentences with at least one quantified metric]
[Scope + future work — 1 sentence, honest about limitations]
```

### Section: Paper Body — Key Sections

> The paper body is only accessible via PDF or HTML view. The arXiv abstract page does not render the body inline. Petra accesses the body sections if she wants to verify the benchmarks in depth. She typically does a quick PDF scroll rather than a full read.

**OBJECT ID:** `arxiv-paper-body`

| Property | Value |
|----------|-------|
| Purpose | Full paper content. Petra skims for benchmarks. Samuel reads sections 3–4 carefully (architecture and physical invariants). |
| Controlled by | Paper content (PhysLink controls all section content and structure). |

#### Introduction Section

**OBJECT ID:** `arxiv-section-introduction`

| Property | Value |
|----------|-------|
| Content requirement | Names the adaptation problem, cites foundational related work (DreamerV3, World Models literature), and states the contribution clearly. Must not bury the contribution in page 4. |
| Petra reading | She scans the first page to confirm the problem framing matches what the README promised. |
| Samuel reading | He reads Introduction carefully for mentions of domain-specific constraints. A pure ML framing of "adaptation" without physics vocabulary is a warning sign. |

#### PhysLink Architecture Section

**OBJECT ID:** `arxiv-section-architecture`

| Property | Value |
|----------|-------|
| Content requirement | Describes the Universal Space API, observation/action space abstraction, and the DreamerV3 adapter layer with a diagram. The architecture diagram must be self-contained enough to understand without reading the full section. |
| Petra reading | She may glance at the architecture diagram to confirm this is a real system, not a theoretical proposal. |
| Samuel reading | He reads this section to understand where physical invariants can be injected. |

#### Physical Invariant Integration Section

**OBJECT ID:** `arxiv-section-physical-invariants`

| Property | Value |
|----------|-------|
| Content requirement | Documents how domain-specific physical constraints (conservation laws, invariants) are registered and enforced in the adaptation loop. Must include at least one worked example (e.g., mass conservation for a robotic system). Must use physics vocabulary correctly. |
| Samuel Priority | **CRITICAL — Samuel's MOMENT OF TRUTH** — this section determines whether he continues or dismisses PhysLink as "ML-only." |
| Petra reading | She notes the existence of this section as evidence that PhysLink addresses scientific rigor. She does not read it in depth. |

#### Experiments & Benchmarks Section

**OBJECT ID:** `arxiv-section-benchmarks`

| Property | Value |
|----------|-------|
| Content requirement | Must include: (1) at least one table comparing PhysLink against a baseline, (2) quantified metrics (adaptation time, success rate, physical constraint violation rate), (3) honest description of experimental conditions, (4) limitations paragraph. |
| Petra Priority | **#5** — she scans for numbers and baselines. A benchmark table with no baseline comparison is a quality failure. |
| Samuel reading | He looks specifically for physical constraint compliance metrics (e.g., "mass conservation violation rate: 0.3%") — this is his primary evidence that physical rigor is measurable. |
| Failure mode | Benchmark section with only loss curves and no interpretable metrics → Petra cannot extract a credible claim for Paula's memo. |

#### Discussion & Limitations Section

**OBJECT ID:** `arxiv-section-discussion`

| Property | Value |
|----------|-------|
| Content requirement | Must acknowledge scope boundaries (e.g., "currently evaluated on robotic domains — extension to fluid dynamics and molecular simulation is planned"), failure modes, and open research questions. |
| Petra reading | Honest limitations increase credibility. A paper with no limitations section feels like marketing copy, not research. |
| Samuel reading | He reads this section to see whether the authors acknowledge the extension to his domain (CFD) or not. An explicit mention of fluid dynamics is a strong GO signal. |

#### GitHub Link in Paper Body

**OBJECT ID:** `arxiv-paper-github-link`

| Property | Value |
|----------|-------|
| Component | Inline URL in paper body (standard academic practice) |
| EN | `Code available at https://github.com/[org]/physlink` |
| Placement | Must appear in the Abstract OR in the Introduction, not buried in Appendix. |
| Samuel Priority | **His primary exit** — he clicks this to navigate from the arXiv paper to the GitHub README. |
| Petra reading | She notes it as confirmation that the code is public and linked. |
| Content requirement | The GitHub URL must be present and correct in the submitted arXiv version. A broken or absent code link is a credibility failure for both personas. |

## Page States

| State | When | Appearance | Impact |
|-------|------|------------|--------|
| Default — preprint available | Paper submitted and indexed on arXiv. | Full abstract page with PDF download. | Standard path for both personas. |
| Pre-submission | Paper not yet submitted. | arXiv page does not exist. | **Critical gap** — both Petra and Samuel cannot access a paper. The arXiv badge on 2.1 would be absent, triggering their NO-GO from the README already. Paper must be submitted before PhysLink public launch. |
| Workshop-accepted | Paper accepted at a workshop or symposium. | Journal-ref field populated on arXiv page. | Positive signal for Petra. Does not change the page structure. |
| Peer-reviewed / journal published | Full paper accepted at conference or journal. | Journal-ref + DOI present. | Strong positive signal — highest credibility for Petra. |
| Author page unavailable | arXiv author profile links are dead. | Author names not clickable or profile empty. | Neutral — Petra reads the affiliation text, not the profile. |
| PDF unavailable | Unusual arXiv error state. | Abstract page loads; PDF link errors. | Petra still forms her assessment from the abstract. She notes the PDF issue; may try later. |
| Stale paper with no revision | Paper submitted > 3 years ago with no revision. | Submission date visible on abstract page. | Risk signal for Petra — she may note "foundational paper, but is the science current?" Not a hard NO-GO. |

## Validation & Errors

All validation is pre-submission / content quality review. No runtime validation.

| Field / Target | Rule | Timing | Error / Recovery |
|----------------|------|--------|------------------|
| Paper title | Must be readable to a non-ML domain scientist. Must name "physical domains" or equivalent. | Pre-submission review. | Revise before arXiv submission. |
| Abstract | Must include problem statement, approach, quantified result, scope. Must include physics vocabulary for Samuel. | Pre-submission review. | Abstract rewrite; target 200 words max. |
| Author affiliations | Must include at least one academic or national research institution. | Pre-submission. | Add academic collaborator or revise affiliation display. |
| GitHub URL in paper | Must be present in Abstract or Introduction. Must be the correct, stable repository URL. | Pre-submission + post-fork check. | Update arXiv with revised version if URL changes. |
| Benchmark table | Must compare against at least one named baseline with quantified metrics. | Pre-submission review. | Add baseline comparison before submission. |
| Physical invariants section | Must use physics vocabulary correctly. Must include at least one worked example. | Pre-submission review with a domain scientist co-reader. | Have a non-ML expert review this section before submission. |
| Limitations section | Must be present and honest. | Pre-submission. | Add limitations before submission — a paper without limitations reads as advocacy, not research. |
| Citation block | arXiv auto-generates — depends on accurate author metadata at submission. | Submission time. | Verify author list and title are correct before submitting to arXiv. |
