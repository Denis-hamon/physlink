## Overview

**Page Purpose:** Petra's **final sustainability check**. She has confirmed license, maintenance, scientific authority, and lab features. The GitHub templates are her last signal: *"Is there a real contributor community maintaining this, or is it a one-person project that will disappear in 18 months?"*

**User Situation:** Petra is building her recommendation memo for Paula. She navigates to the `.github/` folder in the repository and opens the PR template and one or two issue templates. She is reading for structure, not content — does the project expect contributors to follow a process?

**Main User Goal:** Confirm that the project has structured community governance: a PR template that requires changelog updates and test coverage, and issue templates that separate bug reports from feature requests.

**Success Criteria:** PR template requires changelog update + tests. At least 2 issue template types exist (bug report + feature request). Petra forms her final GO and drafts her memo for Paula.

**Entry Points:**
- Navigate to `.github/` folder from repository root (after returning to README from 2.4).
- Click `Contribution guide` from `readme-footer-pr-template-link` (2.1 Footer section).

**Exit Points:**
- Primary: Petra closes the browser tab. Her evaluation is complete — she drafts her recommendation memo for Paula. (Narrative exit — no URL.)
- Secondary: She copies the `.github/` folder URL into her memo as the community governance reference.

## Layout Structure

Two GitHub file views, typically accessed via the `.github/` folder browser.

```text
── .github/ folder view ──────────────────────────────────
  PULL_REQUEST_TEMPLATE.md
  ISSUE_TEMPLATE/
    bug_report.md
    feature_request.md
    domain_extension.md   ← optional, strong positive signal

── PR Template file view ─────────────────────────────────
+----------------------------------------------------------+
| GitHub File Header (native)                              |
| .github / PULL_REQUEST_TEMPLATE.md                       |
+----------------------------------------------------------+
| PR Template Content                                      |
|                                                          |
|  ## Description                                          |
|  ## Type of Change                                       |
|  ## Checklist                                            |
|   - [ ] Tests pass                                       |
|   - [ ] CHANGELOG updated                                |
|   - [ ] Docs updated (if applicable)                     |
|  ## Related Issues                                       |
+----------------------------------------------------------+

── Issue Template file view ──────────────────────────────
+----------------------------------------------------------+
| GitHub File Header (native)                              |
| .github / ISSUE_TEMPLATE / bug_report.md                 |
+----------------------------------------------------------+
| Bug Report Template Content                              |
|                                                          |
|  **Describe the bug**                                    |
|  **To Reproduce** (steps)                                |
|  **Expected behavior**                                   |
|  **Environment** (Python version, PhysLink version)      |
+----------------------------------------------------------+
```

## Typography

Same conventions as `2.2-changelog` — GitHub-rendered Markdown. PR/issue templates use H2 for section headings, bullet lists for checklist items, bold for field labels.

| Element | Markdown | Petra reads as |
|---------|----------|----------------|
| Template section heading | `## Description` | Structural signal — is this template organized? |
| Checklist item | `- [ ] CHANGELOG updated` | Quality gate — is this mandatory or aspirational? |
| Field label | `**Describe the bug**` | UX quality — does the template guide the contributor? |
