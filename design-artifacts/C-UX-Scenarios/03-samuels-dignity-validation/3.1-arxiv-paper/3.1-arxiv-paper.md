## Overview

**Page Purpose:** Samuel's entry point into PhysLink. He arrives cold from a literature search with a domain scientist's skepticism: *"Do these authors understand physics, or are they treating it as a black-box ML problem?"*

**User Situation:** Samuel is a fluid dynamics (CFD) researcher. He fears "physical hallucinations" — ML-generated outputs that violate conservation laws. He will dismiss any tool that speaks only ML language without acknowledging domain constraints.

**Main User Goal:** Validate that the authors speak "Physics," not just "ML." Specifically: find vocabulary that proves the authors understand physical invariants, conservation laws, and the risk of physically inconsistent predictions.

**Success Criteria:** Samuel finds at least two physics vocabulary markers in the abstract or introduction, reads the Physical Invariant Integration section in the PDF, and follows the GitHub link to explore the code. He exits to the GitHub README (3.2 — same URL as 2.1) with the conviction that this project is worth investigating seriously.

**Entry Points:**
- arXiv thematic search (`cs.LG`, `physics.flu-dyn`, `cond-mat`).
- Google Scholar alert for "World Models physics."
- Citation from a related paper.

**Exit Points:**
- Primary: GitHub repository link in the paper body (`arxiv-paper-github-link`) → 3.2-github-readme.
- Secondary: Copies the arXiv URL to share with a physicist colleague.
- NO-GO: Closes the tab. Dismisses PhysLink as "another ML benchmark paper."

## Samuel's Scan Priority

Samuel's reading is slower and deeper than Petra's. He is not doing a credibility audit — he is performing domain validation.

| Step | Object | Signal he's extracting | Pass condition |
|------|--------|------------------------|----------------|
| 1 | `arxiv-paper-abstract` | Do the authors use physics vocabulary? | At least 2 of: `conservation laws`, `physical invariants`, `physical hallucinations`, `domain constraints`, `physical consistency` |
| 2 | `arxiv-paper-affiliations` | Is there a domain scientist (non-ML) among the authors? | Presence of a physicist or domain scientist co-author is a strong bonus; absence is neutral |
| 3 | `arxiv-section-physical-invariants` (PDF) | Is the invariant integration real or superficial? | A worked example with a named physical law (e.g., mass conservation, momentum conservation) |
| 4 | `arxiv-section-benchmarks` (PDF) | Are there physical consistency metrics alongside ML metrics? | At least one metric measuring physical constraint violation (not just prediction accuracy) |
| 5 | `arxiv-section-discussion` (PDF) | Do the authors acknowledge the limits of their physics coverage? | Honest mention of domains not yet covered (e.g., quantum, continuum) |
| 6 | `arxiv-paper-github-link` | Is the code public and accessible? | Absolute GitHub URL present in abstract or introduction |

**GO condition:** Abstract uses physics vocabulary correctly + Physical Invariants section has at least one worked example + GitHub link is present.

**NO-GO conditions (any one is sufficient):**
- Abstract describes the problem purely in ML terms (`latent space`, `world model`, `reward function`) with no physics vocabulary → authors are ML researchers pretending to understand physics.
- Physical Invariants section exists but only describes "soft constraints" in the loss function with no named laws → superficial physics treatment.
- Benchmark section measures only prediction accuracy / adaptation speed with no physical consistency metric → the paper cannot demonstrate its central claim to Samuel.

## Open Questions (Samuel-specific)

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | Is "physical hallucinations" used explicitly in the abstract? | Samuel immediately recognizes this term as evidence the authors understand his fear. It is also a distinctive coined term that differentiates the paper in search results. | 🔴 Open |
| 2 | Is there a CFD (fluid dynamics) worked example in the Physical Invariants section or appendix? | Samuel is a CFD specialist. A fluid dynamics example is the strongest possible alignment signal — it moves him from "this might work" to "this was built for people like me." | 🔴 Open |
| 3 | Should the GitHub link appear in the abstract (not just the introduction)? | Samuel reads the abstract first. If the GitHub link is there, he can navigate to the code immediately without opening the PDF. This also improves the Google Scholar abstract snippet. | 🔴 Open |
