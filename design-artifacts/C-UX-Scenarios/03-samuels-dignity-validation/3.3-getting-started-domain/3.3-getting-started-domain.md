## Overview

**Page Purpose:** Samuel's **MOMENT OF TRUTH**. This page must make him feel that PhysLink was built for people who care about physics — not just ML researchers who tolerate physics as a boundary condition. One concrete, named physical invariant example (mass conservation) is worth more than ten abstract claims.

**User Situation:** Samuel has confirmed the paper is credible (3.1) and found the domain scientist entry point (3.2 in < 10s). He arrives on this page with cautious hope. He is reading actively — not skimming. He will copy the invariant template into his mental model and ask: *"Can I make this work for my CFD data?"*

**Main User Goal:** Find a template for registering a physical invariant (e.g., mass conservation) and confirm he can adapt it to his CFD use case. Then click "Open in Colab" to try it live.

**Success Criteria:** Samuel reads the mass conservation example, understands the `register_invariant()` API call, and clicks "Open in Colab: Domain Scientists Demo." He arrives at 3.4 with enough context to adapt the template to his data.

**Entry Points:**
- `readme-links-domain-scientists-link` from GitHub README (3.2).
- Direct URL from the arXiv paper body reference (some readers may navigate here from the paper).

**Exit Points:**
- Primary: "Open in Colab: Domain Scientists Demo" button → 3.4-colab-validation.
- Secondary: Navigate to `register_invariant()` API reference (docs sidebar).
- Tertiary: Open a GitHub issue using the Domain Extension template (2.5 — Samuel's community return).

> **API Design Example** — This code documents the intended interface, not implementation.

```python
def mass_conservation(state: dict) -> float:
    """
    Returns the mass conservation residual.
    PhysLink minimizes abs(residual) during validation.
    A residual of 0.0 means perfect compliance.
    """
    inflow = sum(state["mass_flow_in"])
    outflow = sum(state["mass_flow_out"])
    return abs(inflow - outflow)
```

#### Invariant Registration

**OBJECT ID:** `domain-invariant-mass-conservation-registration`

| Property | Value |
|----------|-------|
| Component | Code block (Python) |
| EN | See below |

> **API Design Example** — This code documents the intended interface, not implementation.

```python
from physlink import DreamerV3Adapter, register_invariant

adapter = DreamerV3Adapter(obs_space, act_space)

register_invariant(
    adapter=adapter,
    name="mass_conservation",
    fn=mass_conservation,
    tolerance=1e-3,       # acceptable residual threshold
    mode="hard",          # "hard" = reject trajectories that violate; "soft" = penalize
)
```

#### FAQ Items

**OBJECT ID:** `domain-faq-items`

| Property | Value |
|----------|-------|
| Component | Accordion or H3 + body text pairs |
| EN items | See below |

**Q: Can I register multiple invariants simultaneously?**
Yes. Call `register_invariant()` multiple times on the same adapter. Each invariant is checked independently. The compliance report shows each invariant separately.

**Q: My invariant involves tensor operations — can I use NumPy or PyTorch inside the function?**
Yes. The `fn` callable can use any Python library. PhysLink only requires that it accepts a state dict and returns a float.

**Q: What is a "state dict"?**
A Python dict mapping state variable names (e.g., `"mass_flow_in"`, `"velocity"`) to their current values. The exact keys depend on your `ObservationSpace` configuration. See the [ObservationSpace API reference](#).

**Q: Can I use `mode="soft"` to test without rejecting trajectories first?**
Yes. Start with `mode="soft"` to see the violation distribution without blocking results. Switch to `mode="hard"` once you've validated your tolerance threshold.
