## Overview

**Page Purpose:** Samuel's success page. He runs the mass conservation template against his CFD data, sees the compliance graph, and experiences the core PhysLink promise: *speed without sacrificing physical rigor*.

**User Situation:** Samuel arrives with the mass conservation template pre-loaded (from 3.3). He needs to replace the example trajectories with his own CFD data in Cell 3, then run the full notebook. He is nervous — he expects to see a violation somewhere, because every ML tool he has tried eventually does something physically wrong.

**Main User Goal:** Run the compliance validation on his CFD trajectories and see a `PASS` result with 0 violations.

**Success Criteria:** Samuel runs Cell 5 (compliance report) and sees: `mass_conservation: PASS (violations=0/N)`. He sees the compliance graph (histogram showing all residuals within tolerance). He copies the arXiv link and sends it to a physicist colleague. He opens the GitHub Domain Extension issue template (narrative exit).

**Entry Points:**
- `domain-colab-cta-button` from 3.3-getting-started-domain.

**Exit Points:**
- Primary: Shares arXiv link with a colleague (external — email, Slack). Narrative exit.
- Secondary: Opens `.github/ISSUE_TEMPLATE/domain_extension.md` to request CFD-specific support.
- Tertiary: Saves the notebook to his own Google Drive for future use.

## Samuel's Execution Pattern

Unlike Hugo (who executes all cells sequentially without modification), Samuel modifies Cell 3 before running Cell 4 and 5.

| Cell | Samuel's action | Expected duration |
|------|----------------|-------------------|
| Cell 1: Install | Runs without modification. | 30 s |
| Cell 2: Load Template | Reads carefully. Confirms the mass conservation invariant is exactly as documented in 3.3. | 2 min (reading) |
| Cell 3: Load Data | **Modifies** — replaces `example_trajectories` with his CFD trajectory file path or array. | 5–10 min (data prep) |
| Cell 4: Run Adaptation | Runs without modification. Watches the progress bar. | 5–20 min (compute) |
| Cell 5: Compliance Report | Runs. **Reads output carefully.** | 1 min |
| Cell 6: Compliance Graph | Runs. **Studies the histogram.** | 2 min |
| Cell 7: Export | May run to save the compliance report as a PDF or JSON. | 1 min |

**Critical moment:** Cell 5 output. If `violations=0`, Samuel experiences relief and excitement simultaneously. If `violations > 0`, he reads the error message to understand what violated mass conservation — the notebook must make this debuggable, not just show a red cell.

## Page States

| State | When | Samuel's response |
|-------|------|--------------------|
| PASS — violations=0 | All trajectories comply with mass conservation. | **SUCCESS** — Samuel shares arXiv link + opens GitHub issue. |
| PASS — with margin info | violations=0 but some residuals near threshold. | Samuel notes the margin in his export; proceeds with confidence. |
| FAIL — violations > 0 | Some trajectories violate the invariant. | Samuel reads `report.violations()` to understand the failure. He investigates his data (boundary conditions, units, normalization). This is a productive failure — PhysLink caught something real. |
| Install fails | `physlink doctor` returns an error. | Samuel follows the error guidance to fix the runtime. Must be actionable. |
| Data format error | Cell 3 data does not match expected state dict format. | Clear `ValueError` with expected vs. received keys. Samuel corrects his data format. |
| Runtime timeout | Adaptation in Cell 4 exceeds Colab session limit. | Samuel receives a timeout. Guidance: reduce `steps=` or use a smaller trajectory sample for the demo. |

## Validation & Errors

| Field / Target | Rule | Timing | Error / Recovery |
|----------------|------|--------|------------------|
| `example_trajectories` placeholder | Must have the same structure as what `register_invariant()` expects. | Pre-release notebook review. | A physicist domain expert must verify the example data format is realistic. |
| `compliance_report()` output format | Must match `report.summary()` string format documented in 3.3. | Each release, notebook CI. | Auto-test the notebook with `nbmake` or equivalent. |
| Compliance graph | Must render inline in Colab. | Pre-release notebook test. | Test in a fresh Colab runtime before each release. |
| Cell 8 links | All links (arXiv, GitHub, docs) must be correct and live. | Each release. | CI link checker on the notebook. |
| Violation error messages | Must be human-readable for a non-ML user. | Pre-release review. | Have a domain scientist (non-ML) read the error messages and confirm they are actionable. |

## Open Questions

| # | Question | Context | Status |
|---|----------|---------|--------|
| 1 | Should the notebook include a pre-loaded CFD example dataset (synthetic, small, realistic) so Samuel can run the notebook without bringing his own data first? | A working example with realistic CFD data (e.g., a 2D pipe flow with known mass conservation properties) would let Samuel validate the tool before committing his own data. Reduces friction to first success. | 🔴 Open |
| 2 | Should `report.plot()` be interactive (Plotly/Altair) or static (matplotlib)? | Interactive lets Samuel hover over individual trajectories in the histogram. Static is more reproducible for paper figures. | 🔴 Open |
| 3 | Should the compliance report JSON format follow an open standard (e.g., a subset of OpenTelemetry spans) to facilitate integration with lab data management systems? | Some labs (especially well-funded CFD groups) use FAIR data principles and data management platforms. A standard-compatible compliance report would increase institutional adoption. | 🔴 Open |
| 4 | Is the notebook versioned via a GitHub release tag (stable) or the `main` branch (latest)? | Tag-pinned URL is more reproducible for citation in papers. Main branch is always current but may break. Recommendation: tag-pinned URL in docs, with a note pointing to the main branch for the latest version. | 🔴 Open |
